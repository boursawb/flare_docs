Scripted Decorator API

The application programming interface (API) for scripted decorators provides the following methods for writing your code. The first method, which you are required to implement, is called decorate and takes a single parameter matching the context of the decorator. The next method, getDynamicAttributes() is available within every object and returns a hashmap of type <String, Object>. To iterate the various objects, accessor methods will be used to traverse the object hierarchy. Each of the accessor methods are shown below under their associated object. Lastly, the getDynamicAttributes() method includes a get() method and put() method for retrieving the value of a dynamic attribute and creating a new dynamic attribute, respectively.

In general, you should avoid changing lists, only read from them. Whenever possible, you should avoid changing key attributes, as defined in the SDK.

Object specific APIs

The following objects are available within the scripted decorator API.

[IDFBatch] (object = 'batch')

  • getDynamicAttributes(): Map<String, Object>
  • getOrders(): List<IDFOrder>

[IDFOrder] (object = 'order')

  • getDynamicAttributes(): Map<String, Object>
  • getItems(): List<IDFLineItem>

[IDFLineItem] (object = 'lineItem')

  • getDynamicAttributes(): Map<String, Object>
  • getComponents(): List<IDFComponent>

[IDFComponent] (object = 'component')

  • getDynamicAttributes(): Map<String, Object>

You can change attributes at the object context level and they will be saved following the decorate process.

You can read attributes from any object level, for example, line items in a batch, or components contained in line items. However, you should not change attributes on child items, as this may result in errors and will not be saved.
