The proof workflow

includes a feature that allows you to generate a proof of an order before printing the original. Creating a proof will print a single copy of a high quantity order to ensure there are no problems with the printed output. Orders on the products tab can be marked manually as Proof Needed or, an order decorator can be built to automatically identify orders as Proof Needed as they're injected into the system. In order to generate a proof batch, there are two steps you must perform.

  1. Identify which orders require proofing
  2. Create a proof batch for each order identified

For most situations, a decorator will be used to automatically handle this process. As orders flow into the system, a matcher will identify which order should be marked as Proof Required. Once identified, these orders will be tagged with a new proofNeeded attribute and its value will be set to yes. Proof batches can then be created from either the products tab or sort groups tab as required. As it is a less common practice, the option does exist to manually mark an order as Proof Required from the products tab, or even generate a proof batch directly from an order without marking it as such first. For additional information on creating order decorators, see "Order Decorators" on page 1

It is important to understand that a proof is for an order, not a specific item within an order. For example, if an order contains 3 line items for a book, cover and a wrap, a proof batch will result in a single quantity of the entire order. You cannot generate a proof batch for a specific item contained within an order. However, if an item contained within the order has multiple components, you can choose to proof only the components you need by selecting the desired options on the batch dialog. Once the orders have been identified as proof required, a proof batch can be generated from either the products or sort groups tabs. When selecting either of these options to generate a proof batch, will clone the original order, change the quantity to 1 and present the batch dialogue to allow the operator to submit the proof to press. Proof batches are configured to have their own batch ID range, and the batch name can be formatted to meet customer requirements. A proof batch results in two additional attributes being added to the order.

  • Proof (yes / no)
  • Proof Sent Timestamp

The Proof attribute simply identifies the order as either being a proof order or not. The Proof Sent Timestamp attribute will display the date and time that the proof was generated. The Proof Sent Timestamp attribute will be applied to the proof order as well as the original order. All proof batches will be submitted to press with the highest priority of 100 to ensure they are prioritized ahead of other jobs on press.

Proof Required - sort groups

As an order is identified and marked as Proof Required, the value of its proofNeeded attribute will be set to yes. If only proof orders exist in the sort group, the sort group color will change based on the configuration of the system. If, however, there are a mix of orders with differing values for proofNeeded, the appearance of the sort group will not change and there will be no visibility to the fact that the sort group contains both types of orders. For this reason, it is strongly recommended that all split by lists defined in the order sort configuration be updated to include the attribute proofNeeded Once the attribute has been added to all split by lists, the sort groups will dynamically update as orders are identified as Proof Required and those orders being submitted to press as proof batches. The following image shows a sort group which contains only orders that have been identified as Proof Required, as indicated by the color purple.

It is important to understand that orders which have been identified as Proof Required have not been altered in any way, other than having a new proofNeeded attribute set to yes. The order has not yet been cloned or had the quantity changed to 1. Because of this, the Proof Required sort groups can be treated just like a regular sort group, allowing for any of the available batching options to be used. Additionally, there is no limit to the number of times you can proof an order. Each time a proof is generated from an order, a new proof batch will be created. Lastly, does not enforce or require an order that has been marked as Proof Required to result in a proof batch being generated. The decision to generate proof batches from orders tagged as such is up to you.

Sort group color coding

As shown in the above illustration, the system has been configured to display sort groups containing orders marked as Proof Required in purple. This is controlled using two new color override attributes that can designate sort group colors, as well as a reason for the color.

Changing the color of a sort group
  1. Create a new matcher which includes the proofNeeded attribute having a value of yes
  2. Create a decorator for the new matcher and add the attributes colorOverride and colorOverrideReason
  3. Set the workflow and object context for the decorator to sortgroup
  4. Set the value for the colorOverride attribute to any RGB value
Decorator attributes section 
"<attributes>
	<attribute>
		<key>colorOverride</key>
		<type>string</type>
		<value>rgb(127,0,255)</value>
	</attribute>
	<attribute>
		<key>colorOverrideReason</key>
		<type>string</type>
		<value>needs proof</value>
	</attribute>
</attributes>

Proof batches

Once a proof batch is generated, it will be displayed on the batches tab in the same way as other batches. The key difference with proof batches is that they will include special formatting in the batch name and will carry the same additional attributes as previously discussed.

  • Proof
  • Proof Sent Timestamp
